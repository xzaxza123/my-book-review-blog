## 设计目标

分页算法的目标是在不改变文档语义的前提下，根据页面宽高把完整的文章 DOM 拆分成若干物理页面。它既要适配桌面和移动设备不同的尺寸，又要尽量避免出现大块空白或内容被截断的情况。

实现上，算法以 DOM 为基础单元工作，对段落、图片、代码块、表格、列表和引用等元素进行识别和分类，再根据不同类型采用合适的分页策略。

## 输入和输出

分页的输入和输出可以用一段伪代码描述。

```ts
输入
文章容器元素 container
页面宽度 pageWidth
页面高度 pageHeight

输出
若干个页面元素 pages
每个页面元素的高度不超过 pageHeight
```

容器中已经包含了完整的文章内容，包括标题、正文、图片和表格等。分页算法不会修改原始容器，而是创建新的页面元素，并把克隆后的内容放入这些页面中。

## 元素类型识别

在实际拆分页之前，算法会先判断每个元素属于哪一种类型，以决定后续的策略。

```text
段落
图片
代码块
表格
列表
标题
引用块
自定义 HTML
网格容器
```

识别时会充分利用类名和结构信息。例如，mdx-paragraph 包装的段落，如果内部只包含图片容器，则会被识别为图片而不是普通段落。引用块、表格和自定义布局也通过特定的类名和结构进行区分。

## 高度测量与外边距折叠

为了准确判断内容是否能放进一页，算法会使用隐藏的测量容器来计算元素真实占用的高度。这个高度会考虑内边距和外边距，同时通过一个辅助函数模拟外边距折叠。

整体思路可以表示为。

```ts
遍历页面中的子元素
  读取每个元素的高度和上下外边距
  计算当前元素和前一个元素之间的有效间距
  累加到当前页面的总高度
```

这种做法可以更接近浏览器真实的排版行为，避免简单相加带来的误差，从而减少内容刚好被挤出页面一小段的情况。

## 基本分页流程

主流程从内容根节点开始，按顺序处理每一个子元素。

```ts
创建测量根节点
创建第一页
currentHeight 设为 0

遍历内容根的子元素
  克隆当前元素并处理横向溢出
  暂时加入当前页
  重新测量当前页高度
  如果没有超过 pageHeight
    更新 currentHeight
  否则
    尝试压缩上下外边距
    再次测量
    如果仍然超过 pageHeight
      尝试按剩余高度精细拆分
      如果成功拆分出首块
        把首块放入当前页并结束当前页
        处理剩余部分生成新页
      否则
        整块移动到下一页
```

这个流程结合了三个层次的处理方式。先尽量直接放入，再通过轻微调整边距挤出空间，最后才进入更昂贵的精细拆分逻辑。

## 元素级拆分策略

不同类型的元素会采用不同的拆分策略。

### 段落

普通段落在需要拆分时，会先在测量容器中按字符长度做二分查找，找到恰好能放进指定高度的最大子串。然后用这个字数作为分界，将段落拆成两部分。

对于带有富文本或行内标记的段落，会使用保留结构的方式拆分 DOM 节点，而不仅仅是截取纯文本。

### 代码块

代码块默认优先保持完整。高度在一页以内时，直接整体移动到新页。当高度大于一页时，会按比例缩放，使其尽量完整显示在单页中。

在页尾填充场景下，算法允许代码块以保留语法高亮结构的方式拆分为两段。当前页放入首段，剩余部分交给通用代码块策略处理。

### 图片

图片不会被硬拆成上下两段，而是采用整体移动和缩放的策略。

当图片高度小于等于一页时，可以直接整体放入当前页或新页。如果高度超过整页，则会让它独占一页，并按页高进行缩放。对于有说明文字的图片，算法还会适当处理外边距，保证说明文本不会被挤出页面。

在页尾填充模式下，如果通过适度缩放可以让图片完整放入剩余空间，就会以当前页为主显示。如果需要缩放得过小，则宁愿把整张图片移动到下一页。

### 表格

表格的分页以行作为基本单位。

算法首先测量表头的高度，然后在可用高度内尽可能多地放入数据行超出时会将当前可容纳的行集合封装成一张子表，并在末尾加上续表提示。剩余行则作为新的表格，递归按照相同逻辑拆分。

在页尾填充模式下，表格也会先尝试在当前页放入表头和若干行，其余部分交给通用表格拆分逻辑生成后续页面。

### 列表和容器

列表会以列表项为单位进行分页。算法会逐项测量，在不超过页面高度的前提下尽可能放入更多的列表项。对于引用块、自定义 HTML 或网格容器，则采用通用容器拆分逻辑，按子元素组合来决定首块和后续页面。

## 当前页剩余高度填充

主流程中有一个专门用于利用页尾剩余空间的分支。核心思路是只要当前页还有正的剩余高度，就尝试调用对应类型的精细拆分函数，让元素尽可能多地填充当前页。

这一部分逻辑可以抽象成以下伪代码。

```ts
remainingHeight = pageHeight - currentHeight
如果元素可拆分且 remainingHeight 大于 0
  交给专门的按剩余高度拆分函数
  如果返回了首块
    把首块放进当前页
    剩余部分生成新的页面
  否则
    整块移动到下一页
```

为避免留下视觉上几乎看不到的首块，段落和图片等类型在内部还会设置最小占比和最小字符数等门槛，只有首块内容足够多时才会真正拆分。

## 稳健性处理

在工程实践中，分页算法还考虑了几个稳定性问题。

- 在分页前等待图片加载，设置超时时间避免挂起
- 对图片使用 eager 加载策略和解码优化，减少测量误差
- 对各种等待和测量过程设置兜底超时
- 在发现分页后页面仍然过高时输出警告，方便后续排查

这些设计确保了算法在遇到网络波动、懒加载图片或异常内容时，依然可以给出可用的分页结果，而不会阻塞整个阅读体验。

## 与视图层的协作

分页算法本身不关心路由和交互，只关心给定容器和尺寸下如何拆分 DOM。视图层会根据当前设备和布局，决定要传入的 pageWidth 和 pageHeight，并在窗口大小变化时触发重新分页。

最终生成的页面会被书本组件和翻页库消费，添加页脚和装饰后展示给用户。通过这种职责划分，可以在不影响视图层的前提下独立演进分页算法。
