## 开发环境

项目基于 Node.js、npm、Vite 和 TypeScript 开发，推荐使用现代浏览器进行调试。

启动本地开发环境前，请确保已经安装好 Node.js 和 npm。

## 常用脚本

下面的代码块展示了主要命令。

```bash
安装依赖
npm install

本地开发
npm run dev

生产构建
npm run build

预览构建产物
npm run preview
```

开发时通常只需要运行本地开发命令。构建和预览命令主要用于发布前的检查。

## 目录结构回顾

在阅读本指南时，可以结合架构文档一起理解目录结构。

```text
- src
  - core
  - utils
  - hooks
  - view
  - styles
- docs
  - ProjectOverview.mdx
  - DevelopmentGuide.mdx
  - Architecture.mdx
  - PaginationAlgorithm.mdx
  - PaginateAllCases.mdx
```

src 目录下的代码会被打包到最终应用中，docs 目录下的 MDX 文件会在运行时被动态加载并进行分页。

## 代码风格和约定

- 使用 TypeScript 编写业务逻辑
- 使用函数式组件和 React Hook
- 避免在核心分页逻辑中直接操作全局状态
- 在工具模块内保持纯函数风格，便于测试和复用

在添加新模块时，优先考虑是否可以放入 utils 或 core 中。如果和页面场景强相关，则放在 view 层，并尽量保持清晰的输入输出。

## 样式组织方式

项目使用 SCSS 和 PostCSS 管理样式。

```text
- src/styles
  - mdx-styles.css
  - index.scss
```

mdx-styles.css 负责统一控制 MDX 文档中常见元素的视觉风格，例如标题、段落、列表、代码块、表格和引用等。index.scss 则主要处理页面布局、书本容器和主题颜色相关样式。

在新增样式时，优先复用已有的类名和样式约定，避免在文档中直接写行内样式。

## 新增文档的流程

新增一篇文档通常只需要几个步骤。

- 在 docs 目录下添加新的 MDX 文件
- 在文件中写入合适的标题和 frontmatter
- 在首页或导航区域增加入口指向新文档

Vite 插件会在构建和运行时自动扫描 docs 目录，无需手动修改路由配置，只要文件位置和命名合理即可被系统识别。

## 本地调试分页效果

为了验证分页算法在各种内容场景下的表现，可以使用 PaginateAllCases.mdx 文档。这份文档汇集了段落、图片、表格、列表、引用和长代码块等各种情况。

在开发过程中，建议频繁使用这份测试文档观察修改对分页和阅读体验的影响。

## 提交和版本控制

- 保持一次提交集中解决一类问题
- 修改分页算法时，优先在本地写好说明并同步到 PaginationAlgorithm 文档
- 对外发布版本前，至少执行一次完整构建和预览

通过这些约定，可以让这个项目在多人协作和长期维护时都保持清晰和可追踪。
